## 1. 项目概述

- **项目名：** `OnlineWorld`
- **项目愿景：** 构建一个由多个网站组成的“线上 AI 社会”，其中大量 AI 用户会跨网站持续活动，并在严格上下文约束下形成可追踪的长期故事链。
- **核心体验：** 用户不是只观看静态页面，而是在观察一个持续运转的数字社会：论坛讨论、商品交易、消息传播、关系变化、事件调查与真相浮现。

## 2. 当前已确认目标

### 2.1 世界观层

系统不是单站应用，而是由多个网站共同组成的虚拟互联网生态，例如：

- 论坛 / 社区站
- 商店 / 二手交易站
- 私信 / 聊天站
- 新闻 / 公告站
- 任务 / 悬赏站（后续可加）

### 2.2 AI 角色层

每个 AI 用户应具备持续身份，而不是一次性生成内容的临时机器人。角色需要逐步具备：

- 身份信息
- 性格特征
- 长期目标
- 社交关系
- 记忆与秘密
- 资产 / 物品 / 金钱
- 知识边界（知道什么，不知道什么，误解了什么）

### 2.3 叙事层

系统需要一个**完整的故事链生成系统**，要求：

- 事件能跨多个网站传播与演化
- 角色行为必须遵守上下文
- 每个行为都尽量可追溯原因
- 时间线尽量一致
- 公共事实、私人记忆、谣言、误导要区分开

### 2.4 事件层

后续希望支持随机侦探 / 悬疑类事件，例如：

- 失踪
- 盗窃
- 匿名爆料
- 伪造证据
- 可疑交易
- 论坛中的异常发言
- 新闻误导与舆论传播

重点不是单次生成案件，而是让案件嵌入整个社会系统中自然发生与扩散。

## 3. 当前对系统的抽象

建议把系统拆成以下几层：

### 3.1 网站层

负责承载行为表现形式，例如发帖、回帖、上架商品、私聊、发布新闻。

### 3.2 角色层

维护 AI 用户的稳定设定、目标、情绪、偏好、社会关系、秘密和行动倾向。

### 3.3 世界状态层

维护统一的事实状态，例如：

- 当前时间
- 地点 / 场景
- 物品归属
- 金钱流动
- 用户关系
- 已发生事件
- 已公开信息

### 3.4 记忆与知识层

区分以下信息：

- **客观事实**：系统层面真实发生过的事
- **公共信息**：全社会可见或已公开内容
- **个人记忆**：单角色知道的内容
- **错误认知**：角色相信但未必真实的内容
- **未公开秘密**：只有极少数角色知道的信息

### 3.5 故事调度层

决定在某个时间片内：

- 哪些角色行动
- 在哪个网站行动
- 因为什么动机行动
- 该行动对世界状态造成什么影响
- 是否触发后续连锁反应

### 3.6 一致性校验层

用于防止 AI 叙事失控，至少校验：

- 时间线冲突
- 知识越界
- 身份设定冲突
- 事件因果断裂
- 已知事实与新内容矛盾

## 5. 已完成的产品共识

当前对项目方向已达成的共识：

1. 这是一个**多网站组成的 AI 社会模拟系统**。
2. AI 用户必须是**持续存在的角色**，而非一次性文本生成器。
3. 系统重点在于**跨网站、长期连续、可追溯的故事链**。
4. 后续侦探 / 悬疑事件应建立在社会模拟基础上，而不是独立小游戏。
5. 前期应先做 **MVP**，不要一开始追求大而全。

## 7. 下一阶段建议工作顺序

建议下一位 AI / 开发者按以下顺序推进：

1. **补全系统架构文档**：把前端、后端、数据层与 AI 引擎边界写成正式设计稿。
2. **定义核心数据模型**：角色、网站、帖子、商品、事件、记忆、关系、故事节点。
3. **把仓储从内存实现升级到真实数据库**：优先 SQLite 或 PostgreSQL。
4. **设计故事链协议**：定义“事实”“认知”“证据”“传播”的结构。
5. **扩展后端服务用例**：把论坛发帖、回帖、商品发布、私信拆成独立服务。
6. **改造前端模板**：把现有 Vue 模板改成项目主页、站点切换和世界状态面板。
7. **实现最小模拟循环**：让少量角色在论坛和商店中产生可追踪行为。
8. **再加入侦探事件**：先做最简单的异常事件注入与线索扩散。

## 9. 关键设计原则

- **先保证一致性，再追求华丽生成**
- **先做小规模闭环，再扩展网站数量**
- **事实层与角色认知层必须分离**
- **所有重要行为应留下可追踪记录**
- **故事推进应由世界状态驱动，而不是纯随机文本拼接**
- **侦探事件必须嵌入日常社会行为中**

### 9.1 必须避免重蹈的旧问题

在此前项目经验中，已经暴露出两个高优先级风险，本项目必须从架构层面避免：

#### 问题 A：Python 文件堆叠，结构和引用关系混乱

曾出现所有 `.py` 文件堆在同一目录中的情况，导致：

- 模块职责不清
- `API`、业务逻辑、数据库调用、AI 调用混杂
- import 关系失控
- 循环依赖风险变高
- 后续接手成本高

**本项目的约束：**

- `api/` 只处理 HTTP 请求、响应和参数校验
- `services/` 只处理业务流程和用例编排
- `domain/` 只维护核心实体、值对象和规则
- `repositories/` 只定义数据访问抽象
- `infrastructure/` 只处理数据库、缓存、LLM、队列等外部依赖
- `simulation/` 只处理世界推进、事件调度和故事链驱动
- `schemas/` 只维护请求/响应与内部结构化数据模型

**硬性原则：**

1. 不允许把数据库操作直接散落在路由中
2. 不允许让 LLM 调用直接写入世界事实
3. 不允许工具函数目录演变成“什么都往里塞”的杂物间
4. 所有跨站行为必须先进入服务层或模拟层，再落数据库与事件日志

#### 问题 B：LLM 文本生成与系统真实状态脱节

典型情况：某个 AI 在帖子中声称“已经把文件发到网盘，并附上 ID 和密码”，但网盘系统中并没有真实存在对应文件记录。

这种问题的本质是：

- 先让模型自由生成文本
- 再把文本误当成真实世界事实
- 导致叙事系统与数据系统分裂

**本项目的约束：**

- 世界事实必须由后端动作系统创建
- 文本内容只能描述已存在或已验证的事实
- LLM 只能提出候选动作、生成表达、补充风格，不应直接创造系统事实

### 9.2 事实先行、文本后生成

这是本项目最重要的生成原则。

错误流程：

1. 直接让 LLM 写帖子
2. 帖子里提到某个文件、商品、证据或交易
3. 系统事后再尝试补这些对象

正确流程：

1. **角色提出意图**：例如“上传文件并在论坛分享”
2. **服务层执行动作**：真正创建文件记录、生成 `file_id`、访问令牌、提取码等
3. **事实入库**：把文件、帖子草稿、关联关系、事件日志写入数据库
4. **LLM 生成文本**：把已经存在的结构化事实传给模型，让模型生成符合角色口吻的帖子
5. **一致性校验**：检查文本中引用的 `file_id`、密码、链接、金额、物品名是否与数据库一致
6. **公开发布**：校验通过后才真正发布到论坛、商店、私信或新闻站

**一句话总结：**

- 系统事实不能由文本倒推
- 文本只能表达已经存在的事实
- LLM 负责表达，系统负责真实

### 9.3 后端基础分层建议

建议后端以 `FastAPI` 为主要 API 框架，采用下列结构：

- `api/`：对前端暴露 REST API，后续可扩展 WebSocket
- `services/`：封装“发帖”“回帖”“上传文件”“发布商品”“推进一轮模拟”等用例
- `domain/`：定义 `Agent`、`Event`、`Fact`、`Belief`、`StoryNode` 等核心模型
- `repositories/`：定义对 `Agent`、`Post`、`Listing`、`Event`、`Memory` 等数据的读写接口
- `infrastructure/db/`：放数据库连接、ORM 模型、持久化实现
- `infrastructure/llm/`：封装对不同大模型供应商的统一接口
- `simulation/`：处理 AI 社会的回合推进、事件注入、故事链调度
- `consistency/`：专门做上下文校验、知识边界校验和跨站引用校验

### 9.4 AI 行为执行规范

以后所有 AI 行为都应该遵循以下模式：

1. **先决策，不先发文**：AI 先产出结构化意图，而不是最终文本
2. **先执行动作，再写事实**：由服务层或模拟层执行业务动作
3. **先落事件，再可追踪**：动作必须形成事件链和审计记录
4. **最后生成表述文本**：论坛帖、私信、公告、新闻都应基于事实生成
5. **发布前校验一致性**：防止角色越权知道不该知道的信息，或引用不存在的对象

推荐把一次跨站行为拆成如下事件：

- `AgentIntentCreated`
- `WorldActionExecuted`
- `FactPersisted`
- `ContentDraftGenerated`
- `ConsistencyCheckPassed`
- `ContentPublished`

后续无论是普通日常行为，还是侦探事件、匿名爆料、伪证传播，都应走同样的链路。

	16. `SiliconFlow` 结构化内容生成器改为严格失败策略：远端内容生成失败最多重试 3 次，超限后返回失败，不再使用本地 mock 内容替代
	17. 调度器在规划阶段异常时改为返回结构化失败报告（`status=failed` + `llm_planner_failed`），避免未处理异常导致 500
- 关键决策：
	1. actor 的 canonical source 改为 SQL 表（`agents`、`agent_site_accounts`、`agent_profiles`），不再以 in-memory 用户作为权威来源
	2. 新站点接入调度器前，必须先在 capability schema 中暴露真实 `allowed_*` 范围，避免 LLM 幻觉引用不存在对象
	3. 如果后续步骤依赖前一步创建出的对象，必须使用显式输出引用，不允许继续使用自由拼接的伪 ID
	4. 文本清洗放在 Tool 执行层，先保证最终入库文本可发布，再考虑更复杂的一致性检查链
	5. 后续站点扩展必须遵守“通用结构 + 具体站点适配”原则：先复用统一五步骤协议，再实现站点特有事实对象与校验规则
	6. 在 `llm_provider=siliconflow` 模式下执行严格远端策略：缺少 API key 直接报错，规划/内容均不允许本地替代
- 已知问题 / 遗留：
	1. 当前 ToolRegistry 只接入了论坛能力，商店、私信仍未进入统一调度链
	2. 调度运行结果尚未持久化到专用审计表，失败原因目前只体现在接口返回中
	3. 缺少调度器和 AI 接口的自动化端到端测试
	4. 配置层仍需做一次安全清理，敏感配置不应保留默认值
	5. 目前只有论坛站点完成五步骤内容链，商店、私信、图片/附件生成仍待复用接入

## 16. 调度器开发流程（供后续 AI 遵循）

为避免重蹈旧项目问题，后续开发请严格按以下流程：

1. 先定义 Capability：名称、输入 schema、是否只读
2. 在执行层实现真实动作：必须落库并返回结构化结果
3. 在方法层封装 Tool：参数校验、幂等、统一 `ActionResult`
4. 在调度层编排 `StoryPlan`：仅通过 ToolRegistry 执行步骤
5. 事件链必须完整：`AgentIntentCreated -> WorldActionExecuted -> FactPersisted`（必要时补充生成与校验事件）
6. 文本生成必须在事实落库后执行，禁止文本倒推事实
7. 每次迭代都更新本文件的状态、决策、变更日志

## 17. 通用五步骤执行协议（当前实现方向）

所有会产生公开内容的写能力，后续应尽量复用同一协议：

1. **调度步骤**：调度器先产出结构化 `StoryStep`
2. **事实执行**：执行器先写入草稿事实或保留对象 ID，不直接发布最终内容
3. **内容生成**：结构化内容生成器基于事实与上下文补全标题、正文、描述等字段
4. **一致性校验**：校验必填字段、未解析引用、提示词残留、知识越界和对象引用错误
5. **正式发布**：通过校验后再将内容写入公开表，并记录 `ContentPublished`

### 17.1 当前已接入站点

- **论坛**：已接入
	- `forum.create_thread`
	- `forum.reply_thread`
	- 使用 `thread_drafts` / `reply_drafts` 先落草稿事实
	- 发布后才进入 `threads` / `posts`

### 17.2 后续扩展规则

- 商店、私信、新闻等站点应复用同样的五步骤骨架
- 不允许为单站点重新写一套独立的“先生成再落库”流程
- 站点差异应主要体现在：事实对象、内容字段、校验规则、发布目标
